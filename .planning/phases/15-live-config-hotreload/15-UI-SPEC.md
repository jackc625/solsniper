# Phase 15 — UI Design Contract

> Visual and interaction contract for the frontend surface of the config hot-reload phase. Generated by gsd-ui-researcher, verified by gsd-ui-checker.
>
> **Scope note:** Phase 15 is primarily a backend refactoring phase. The frontend surface is limited to three changes: (1) a new CONFIG_CHANGED feed card in the Live Feed, (2) two new Settings fields (Poll Interval, Buy Slippage), and (3) adding `CONFIG_CHANGED` to the SSE event listener. All visual decisions inherit the Phase 13 design system established in `dashboard/index.html`.

## Component Inventory

### New: CONFIG_CHANGED Feed Card

The CONFIG_CHANGED event appears in the Live Feed as a standard `FeedCard` component. No new component is created -- the existing FeedCard receives new entries in its lookup maps.

**BADGE_COLORS addition:**
```
CONFIG_CHANGED: 'var(--amber)'
```

**EVENT_LABELS addition:**
```
CONFIG_CHANGED: 'CFG'
```

**Visual behavior:**
- Left border: 3px solid var(--amber) (matching badge color, per existing FeedCard pattern)
- Badge: Amber background (#f0a500), black text (#000), text "CFG", 0.65rem font, 700 weight
- Mint column: Empty string -- shows no mint link, no Solscan link, no pump.fun link
- Detail text: Shows the `detail` field from the event payload (e.g., "Settings updated: stopLossPct, buyAmountSol")
- Right column: Timestamp only -- no P&L, no safety score, no buy amount
- Expand panel: Shows `detail` field only (no mint, no links, no source)
- isDryRun badge: Not shown (config changes are not trade-specific)

**Feed card dimensions (unchanged from existing):**
- Collapsed height: ~28px (auto from single-row grid)
- Padding: 0.35rem 0.75rem 0.35rem 0.6rem
- Margin-bottom: 2px between cards

## Interaction Contract

### Settings Save Flow (existing, extended)

1. User modifies one or more fields (existing fields + new Poll Interval and Buy Slippage)
2. User clicks "APPLY CHANGES" button
3. Button text changes to "WRITING...", opacity 0.6, cursor not-allowed
4. POST /api/config with patch payload
5. **On success:**
   - `configSignal.value` updates with new config (existing)
   - "SAVED [checkmark]" text appears in teal (var(--teal)) for 2000ms (existing)
   - CONFIG_CHANGED event emitted via SSE (NEW)
   - CONFIG_CHANGED feed card appears in Live Feed (NEW)
6. **On error:**
   - "ERR: {message}" text appears in red (var(--red)) (existing)
   - No CONFIG_CHANGED event emitted

### CONFIG_CHANGED Feed Card Interaction

- Click to expand: Reveals detail panel with full detail text (same expand/collapse pattern as all FeedCard instances)
- Hover: Background transitions to #1f1f1f (existing FeedCard hover behavior)
- No external links (no mint to link to)
- Auto-scroll: Card triggers auto-scroll if isLive is true (existing LiveFeed behavior)

### New Input Fields Interaction

- Focus: 1px solid var(--amber) outline, outline-offset 1px, border-color var(--amber) (existing global input:focus rule)
- Type validation: Browser-native `min`/`max`/`step` constraints on number inputs
- Value persistence: Values read from `configSignal.value` on mount and after successful save
