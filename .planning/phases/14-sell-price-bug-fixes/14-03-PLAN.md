<objective>
Fix the two dashboard SQL queries that compute P&L incorrectly. The history endpoint and stats endpoint both use `sell_price_sol - buy_price_sol` (per-token unit delta), but per the locked decision, the correct formula is `sell_price_sol - amount_sol` (total SOL out minus total SOL in). Also fix the win rate denominator to exclude legacy trades with NULL sell_price_sol.

Purpose: Without this fix, even after sellers correctly store sell_price_sol, the dashboard would display wrong P&L numbers because buyPriceSol is a per-token unit price while sellPriceSol is total SOL received.
Output: Dashboard shows accurate per-trade and aggregate P&L.
</objective>

**1. Fix the history endpoint SQL** (`GET /api/trades/history`):

Change the pnl_sol computation in the SELECT:
```sql
-- OLD (BROKEN): sell_price_sol - buy_price_sol
-- NEW (FIXED): sell_price_sol - amount_sol
CASE WHEN sell_price_sol IS NOT NULL AND amount_sol IS NOT NULL
     THEN sell_price_sol - amount_sol ELSE NULL END as pnl_sol
```

The full query becomes:
```sql
SELECT id, mint, state, source, amount_sol, buy_price_sol, sell_price_sol,
       created_at, updated_at, dry_run,
       CASE WHEN sell_price_sol IS NOT NULL AND amount_sol IS NOT NULL
            THEN sell_price_sol - amount_sol ELSE NULL END as pnl_sol
FROM trades
WHERE state IN ('COMPLETED', 'FAILED', 'ABANDONED')
  AND (dry_run IS NULL OR dry_run = 0)
ORDER BY updated_at DESC
LIMIT 500
```

**2. Fix the stats endpoint SQL** (`GET /api/stats`):

Change the total_pnl_sol computation:
```sql
-- OLD (BROKEN): sell_price_sol - buy_price_sol
-- NEW (FIXED): sell_price_sol - amount_sol
SUM(CASE WHEN sell_price_sol IS NOT NULL AND amount_sol IS NOT NULL
         THEN sell_price_sol - amount_sol ELSE 0 END) as total_pnl_sol
```

**3. Fix the win rate denominator:**

Add `completed_with_pnl` and `total_with_pnl` to the stats SQL:
```sql
SELECT
  COUNT(*) as total,
  SUM(CASE WHEN state = 'COMPLETED' THEN 1 ELSE 0 END) as completed,
  SUM(CASE WHEN state = 'FAILED' OR state = 'ABANDONED' THEN 1 ELSE 0 END) as failed,
  SUM(CASE WHEN sell_price_sol IS NOT NULL AND amount_sol IS NOT NULL
           THEN sell_price_sol - amount_sol ELSE 0 END) as total_pnl_sol,
  SUM(CASE WHEN state = 'COMPLETED' AND sell_price_sol IS NOT NULL
           AND sell_price_sol - amount_sol > 0 THEN 1 ELSE 0 END) as wins,
  SUM(CASE WHEN sell_price_sol IS NOT NULL THEN 1 ELSE 0 END) as total_with_pnl
FROM trades WHERE state IN ('COMPLETED','FAILED','ABANDONED')
  AND (dry_run IS NULL OR dry_run = 0)
```

Update the TypeScript type cast to include the new fields:
```typescript
as { total: number; completed: number; failed: number; total_pnl_sol: number; wins: number; total_with_pnl: number }
```

Update the win rate computation:
```typescript
// OLD: winRate = completed / total (includes legacy NULL rows as losses)
// NEW: winRate = wins / total_with_pnl (only trades with sell_price_sol data)
const winRate = completedRow.total_with_pnl > 0
  ? Math.round((completedRow.wins / completedRow.total_with_pnl) * 100)
  : 0;
```

The response shape stays the same (winRate is still a 0-100 number), so no frontend changes needed.

**Key detail on "wins" definition:** A win is a COMPLETED trade where `sell_price_sol - amount_sol > 0` (positive P&L). This is more accurate than the old denominator which counted all terminal trades. Legacy trades with NULL sell_price_sol are excluded from both numerator and denominator.
