**BUG 1 — jito-seller.ts:152-165** (Claude's discretion on interval/backoff):
Replace the single-request `pollBundleStatus` with a polling loop using exponential backoff. The function is already wrapped in SellLadder's `Promise.race` with `jitoTimeoutMs` (30s default), so it does NOT need its own max-attempts counter or internal timeout.

Implementation:
```typescript
async function pollBundleStatus(bundleId: string): Promise<'Landed' | 'Failed' | 'Pending'> {
  const INITIAL_DELAY_MS = 1000;
  const MAX_DELAY_MS = 5000;
  let delay = INITIAL_DELAY_MS;

  // Loop indefinitely -- SellLadder's Promise.race timeout is the outer bound
  while (true) {
    await new Promise(resolve => setTimeout(resolve, delay));

    const response = await fetch(JITO_BUNDLE_URL, {
      method: 'POST',
      headers: { 'Content-Type': 'application/json' },
      body: JSON.stringify({
        jsonrpc: '2.0',
        id: 1,
        method: 'getBundleStatuses',
        params: [[bundleId]],
      }),
    });
    const json = await response.json();
    const status = (json?.result?.value?.[0]?.confirmation_status as string) ?? 'Pending';

    if (status === 'Landed' || status === 'Failed') {
      return status as 'Landed' | 'Failed';
    }

    // Backoff: 1s -> 2s -> 4s -> 5s (capped)
    delay = Math.min(delay * 2, MAX_DELAY_MS);
  }
}
```

The `while(true)` is safe because the SellLadder timeout (jitoTimeoutMs) will always abort this function. The comment at line 138-139 already documents the intent: "Poll for bundle landing (timedout by SellLadder via Promise.race)". Do NOT add a separate internal timeout — see Pitfall 3 in RESEARCH.md.

Update jito-seller.test.ts to test the polling loop: mock `fetch` to return 'Pending' on first call(s), then 'Landed' on a subsequent call. Verify the function makes multiple requests and returns 'Landed'. Also test 'Failed' terminal state.
