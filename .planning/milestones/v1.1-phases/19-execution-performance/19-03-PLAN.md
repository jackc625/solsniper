**Step 3: Insert balance check in detection handler**

In the `detectionManager.on('token', ...)` handler, insert the balance check AFTER the max-concurrent-positions check (line 173) and BEFORE `safetyPipeline.evaluate(event)` (line 175). Per D-14, this saves wasted safety check RPC calls when balance is insufficient.

Insert this block between the max-positions check and the safety pipeline call:
```typescript
      // EXE-12: Balance guard -- skip buy if wallet SOL is below threshold (D-14: before safety pipeline)
      const cfg = getRuntimeConfig();
      const balanceCheck = await balanceGuard.check(
        rpcManager.getConnection(),
        new PublicKey(walletPubKeyStr),
        cfg.buyAmountSol,
        cfg.minBalanceBufferSol,
      );
      if (!balanceCheck.sufficient) {
        // D-15: Skip buy and emit LOW_BALANCE event for dashboard SSE
        botEventBus.emit('event', {
          type: 'LOW_BALANCE',
          mint: event.mint,
          ts: Date.now(),
          detail: `balance ${balanceCheck.balanceSol.toFixed(4)} SOL < threshold ${balanceCheck.thresholdSol.toFixed(4)} SOL`,
        });
        log.warn(
          { mint: event.mint, balanceSol: balanceCheck.balanceSol, thresholdSol: balanceCheck.thresholdSol },
          'Balance below buy threshold -- skipping safety checks',
        );
        return; // D-19: buys only -- sells never blocked
      }
```

Note: The existing code already calls `getRuntimeConfig()` later in the handler (lines 177, 195, etc.). The new `cfg` variable can either be reused by the downstream code (rename existing references), or use a separate const here. The cleanest approach: move the `getRuntimeConfig()` call to the beginning of the handler (before the max-positions check) so all downstream code shares one config snapshot per event. This also ensures `maxPositions` uses runtime config (it already does on line 169).

**Step 4: Invalidate balance cache after successful buy**

After `void executionEngine.buy(event)` (line 201), the buy is fire-and-forget. The balance cache will naturally expire after 5s. For extra safety with concurrent detections, add a cache invalidation. However, per Pitfall 4 in RESEARCH, maxConcurrentPositions=1 prevents concurrent buys. The 5s cache TTL is sufficient. No explicit invalidation needed at this point.
