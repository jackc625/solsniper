<objective>
Build a standalone Node.js audit script that analyzes safety pipeline accuracy by correlating historical safety decisions with actual trade outcomes. Produces a Markdown report with false positive/negative rates, per-check accuracy, and weight/threshold recommendations.

Purpose: SAF-10 requires quantified audit of current safety pipeline accuracy. SAF-11 requires calibration recommendations based on that audit. This script is the data foundation for informed weight adjustments -- no auto-apply, user reviews recommendations and adjusts manually (per D-06).

Output: `scripts/audit-safety.ts` executable via `npx tsx scripts/audit-safety.ts --db data/trades.db --logs bot.log`, produces `reports/safety-audit-YYYY-MM-DD.md`.
</objective>

  <action>
    Create `scripts/audit-safety.ts` (per D-01, D-03, D-04, D-06):

    **CLI interface:**
    ```typescript
    // Usage: npx tsx scripts/audit-safety.ts --db data/trades.db [--logs bot.log] [--output reports/]
    // --db:     Path to SQLite trades database (required)
    // --logs:   Path to pino JSON log file (optional -- if absent, only uses DB safety columns)
    // --output: Directory for report output (default: reports/)
    ```
    Parse args with process.argv (no external arg parser needed for 3 flags).

    **Module structure (exported functions for testability):**

    1. `parseLogLine(line: string): SafetyLogEntry | null`
       - JSON.parse each line, return null if not a safety-pipeline module entry
       - Extract: `{ mint, source, decision, aggregateScore, minSafetyScore, rejectionReasons, tier1, tier2, tier3, durationMs, time }`
       - Interface: `SafetyLogEntry` with those fields

    2. `readLogFile(logPath: string): Promise<SafetyLogEntry[]>`
       - Use `readline.createInterface` with `fs.createReadStream` to read line-by-line
       - Call `parseLogLine` per line, filter nulls
       - Return array of parsed entries

    3. `queryTradesFromDb(dbPath: string): TradeAuditRow[]`
       - Open DB read-only with `better-sqlite3`
       - Query completed trades with P&L:
         ```sql
         SELECT mint, source, amount_sol, sell_price_sol,
                (sell_price_sol - amount_sol) AS pnl_sol,
                safety_score, safety_rejection_reasons, safety_checks_detail,
                created_at
         FROM trades
         WHERE state = 'COMPLETED' AND sell_price_sol IS NOT NULL
         ```
       - Also query all trades (including FAILED/ABANDONED) for loss analysis
       - Return typed rows

    4. `correlateTradesWithDecisions(trades: TradeAuditRow[], decisions: SafetyLogEntry[]): CorrelatedEntry[]`
       - Match by mint (trades only exist for PASSED tokens)
       - Mark each trade as: profitable (pnl > 0), loss (pnl <= 0)
       - For unmatched decisions (REJECTED in logs, no trade): these are the false positive candidates

    5. `sampleRejectedMints(decisions: SafetyLogEntry[], count: number): string[]`
       - Filter decisions where `decision === 'REJECTED'`
       - Random sample up to `count` mints (default 50, per D-03)
       - Return mint addresses for optional price checking

    6. `computeStats(correlated: CorrelatedEntry[], rejectedSample: string[]): AuditStats`
       - Calculate: total passed, total rejected, profitable count, loss count
       - P&L summary: total SOL gained, total SOL lost, net P&L
       - False negative rate: rejections where token later pumped (from sample check)
       - Per-check accuracy: for each check source (rugcheck, holder, creator), count how many times it contributed to a correct rejection vs incorrect rejection
       - Score distribution: bucket scores into 0-10, 10-20, ..., 90-100 ranges
       - Average duration of safety evaluations
       - Weight/threshold recommendations based on correlations (per D-06):
         - If check X has high false rejection rate, recommend lowering its weight
         - If minSafetyScore is too high (most rejections are near threshold), recommend lowering
         - If minSafetyScore is too low (most losses come from barely-passed tokens), recommend raising

    7. `generateReport(stats: AuditStats, outputDir: string): string`
       - Create Markdown report with sections:
         - **Summary**: Total tokens evaluated, pass/reject counts, date range
         - **Trade Outcomes**: P&L breakdown for passed tokens
         - **Per-Check Accuracy**: Table showing each check's contribution to correct/incorrect decisions
         - **Score Distribution**: Histogram of aggregate scores
         - **False Positive Estimate**: Results from rejected mint sampling
         - **Recommendations**: Data-driven suggestions for weight/threshold adjustments
       - Write to `{outputDir}/safety-audit-{YYYY-MM-DD}.md`
       - Return the file path

    **Main function:**
    ```typescript
    async function main() {
      const args = parseArgs();
      const trades = queryTradesFromDb(args.db);
      const decisions = args.logs ? await readLogFile(args.logs) : [];
      const correlated = correlateTradesWithDecisions(trades, decisions);
      const rejectedSample = sampleRejectedMints(decisions, 50);
      // Note: actual price fetching for rejected mints is a future enhancement.
      // For now, log the sampled mints for manual checking.
      const stats = computeStats(correlated, rejectedSample);
      const reportPath = generateReport(stats, args.output);
      console.log(`Audit report written to: ${reportPath}`);
    }
    ```

    **Important notes:**
    - Open SQLite database as `{ readonly: true }` -- audit script must NEVER write to the live DB
    - Use `createRequire(import.meta.url)` for better-sqlite3 ESM interop (same pattern as trade-store.ts)
    - Handle case where no log file provided (use only DB safety columns from Plan 01 migration)
    - Handle case where DB has no completed trades (report "insufficient data")
    - Handle case where safety columns are all NULL in existing trades (pre-Phase 18 trades)

    **Test file** `scripts/audit-safety.test.ts`:
    Test the pure functions (parseLogLine, correlateTradesWithDecisions, computeStats, generateReport) with mock data. Use in-memory SQLite for queryTradesFromDb tests. Do NOT test readLogFile (filesystem-dependent). Create mock SafetyLogEntry and TradeAuditRow objects.
